# Implementation Plan: DailyPlanly MVP Platform

**Branch**: `002-dailyplanly-mvp-platform` | **Date**: 2026-05-28 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/002-dailyplanly-mvp-platform/spec.md`

## Summary

Build the DailyPlanly MVP from a static coming-soon page into a full browser-based
productivity platform. The platform delivers four core product pillars: Planner Systems,
Checklist Systems, Template Gallery, and Productivity Utility Tools — all running
entirely in the browser with local-first storage and no backend dependency.

The technical approach uses Next.js 15 (App Router, static export), TypeScript strict,
Tailwind CSS v4, Zustand, idb (IndexedDB), `@dnd-kit` for drag-and-drop,
`@react-pdf/renderer` for PDF export, and `html-to-image` for PNG/JPG export.
All heavy features are lazy-loaded.

## Technical Context

**Language/Version**: TypeScript 5.5 strict, Node.js 20

**Primary Dependencies**:
- Next.js 15 (App Router, `output: 'export'`)
- React 19
- Tailwind CSS v4
- shadcn/ui (copied primitives)
- Zustand 5
- idb 8 (IndexedDB wrapper)
- @dnd-kit/core + @dnd-kit/sortable (drag-and-drop)
- @react-pdf/renderer 4 (lazy-loaded)
- html-to-image (lazy-loaded, PNG/JPG export)
- next-themes (dark/light)
- next-sitemap

**Storage**: IndexedDB via idb (planners, checklists) + localStorage (recents, settings)

**Testing**: Vitest (unit/integration), Playwright (E2E)

**Target Platform**: Modern browsers (Chrome 120+, Firefox 120+, Safari 17+), desktop-first

**Project Type**: Static web application (Next.js static export → CDN deployment)

**Performance Goals**:
- First Contentful Paint < 1.5s on broadband
- Initial JS bundle < 80kb (PDF/image export engines lazy-loaded separately)
- Theme switch: instant (CSS variable swap, no re-render)
- Export PDF < 4s for single-page planners

**Constraints**:
- Zero server-side infrastructure (static export)
- No authentication, no cloud storage, no WebSockets
- Fully offline after initial load
- All exports client-side only

**Scale/Scope**:
- ~30 templates at launch across 8 categories
- ~10 utility tools across 3 initial categories
- 9 planner block types
- Single developer, incremental delivery

## Complexity Tracking

| Justified Complexity | Why Needed | Simpler Alternative Rejected Because |
|---|---|---|
| Block registry pattern | 9 block types at launch; needs to be extensible without modifying layout engine | Switch-case monolith becomes unmaintainable at 5+ types |
| @dnd-kit drag-and-drop | Block reordering (planner) + item reordering (checklist) must be accessible and touch-friendly | HTML5 DnD: no touch support, no keyboard accessibility |
| html-to-image | PNG/JPG export is explicitly required by FR-007 and FR-014 | No simpler browser-native path for styled DOM-to-image |
